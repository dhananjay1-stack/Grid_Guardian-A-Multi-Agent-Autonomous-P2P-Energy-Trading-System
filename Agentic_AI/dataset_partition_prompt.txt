PROMPT: Implement `dataset_partition.py` — Validate, Clean, and Partition the Grid-Guardian Dataset

Brief goal (one line)
Build `dataset_partition.py`, a single reproducible Python tool that fully **audits, cleans, feature-checks, documents**, and **splits** the Grid-Guardian dataset into train / val / test sets while preserving temporal and structural properties and guaranteeing representation of rare safety events. Use this prompt to generate production-ready code (CLI, tests, and outputs).

Reference: The requirements follow the post-dataset RL pipeline guidance (Step 1 — *Validate and Partition the Dataset*).


Exact input assumptions (canonical schema)

* Input must be the canonical per-timestep-per-household CSV with header (23 cols):
  `ts,household_id,pv_gen_kw,load_kw,net_kw,soc_kwh,soc_capacity_kwh,battery_power_kw,price_signal,forecast_irradiance_1h,forecast_irradiance_3h,forecast_temp_1h,actual_irradiance_wm2,voltage_v,current_a,offer_id,offered_kwh,offer_price,trade_id,commit_hash,event_flag,reward,safety_violation`
* If header differs, script must fail early (or report mapping in `--dry_run`).

High-level workflow (implement exactly these stages)

1. **Provenance & sanity header check (mandatory)**

   * Load input header (stream if large). Verify exact header & column count. Record file checksum and metadata (file size, row count, first/last ts). Write `provenance.json`.

2. **Lightweight streaming schema + parse pass**

   * Stream through dataset in chunks (configurable chunk size). For each chunk:

     * Parse `ts` to timezone-aware UTC; coerce numeric fields; record parse failures.
     * Build lightweight index: unique households, row counts per household, earliest/latest ts per household.
   * Abort with `critical` if parse failure rate > 0.5% (configurable).

3. **Completeness & missingness audit**

   * Compute per-column missing counts & fractions. Write `missing_summary.csv` and `missing_matrix.png` (if `--visualize`).
   * For columns with missing fraction > `--max_missing_threshold`, mark as `review_required`. If `--strict` set, treat as failure.
   * Document gaps by household: find contiguous missing-timestep ranges and largest gap; produce `gaps_summary.csv`.

4. **Imputation strategy & implementation**

   * Offer `--impute_method` modes:

     * `drop`: drop rows with required missing values (configurable required cols).
     * `interpolate`: time-based linear interpolation inside daylight/household windows; limit gap-fill to ≤ 30 minutes default.
     * `model`: train simple per-household regression (e.g., time-of-day + irradiance → load) to impute longer gaps (only used if `--impute_method=model`). Save model artifacts to `out_dir/models/`.
     * `flag`: do not fill; instead add boolean `<col>_imputed` flags and leave nulls for later.
   * Imputation must not change original file; write a cleaned copy `generated_dataset_cleaned.csv.gz` (or in-memory if `--dry_run`).

5. **Feature & distribution checks (critical for safety)**

   * For each numeric feature compute min/mean/median/max/std and check against realistic ranges (configurable per column in `partition_config.yaml`). Examples: `actual_irradiance_wm2 >= 0 && <= 1500`, `temperature_C ∈ [-60,60]`, `soc_kwh ∈ [0, soc_capacity_kwh]`. Flag and list rows violating physical bounds.
   * Detect negative irradiance and clip to 0 (or flag if above a tolerance). Log any automatic clipping in `fix_log.json`.
   * Outlier detection: implement `iqr`, `zscore`, and `mad` options; produce `outliers.csv` and optional robust winsorization if `--impute_method==interpolate && winsorize=true`.

6. **Action / coverage & bias checks**

   * Verify that action space is covered: distribution of `battery_power_kw` (charge/discharge), `offer_id` creation frequency, and action counts per household. Compute per-household action histograms and global action coverage.
   * Detect bias: e.g., if one archetype never sells (0 offers), flag as `policy_bias`. Check that majority of households are not under-represented (min rows per household threshold).
   * Ensure representation of rare events: count `event_flag` occurrences (normal, cloud_ramp, grid_outage, ev_arrival). If any event intended to be reserved for test has fewer than `--min_event_count` occurrences, raise warning/error.

7. **Distributional shift & stationarity checks**

   * Compute rolling monthly/weekly statistics for key features (irradiance, load, pv_gen) and perform KS-test / Wasserstein distance between early and late windows to detect drift. Output `drift_report.json` and plots if `--visualize`.
   * If strong non-stationarity detected, recommend time-based splitting and domain-randomization in later training.

8. **Splitting strategies (implement multiple, configurable)**
   Provide exact implementations for these strategies — user chooses via `--strategy`:

   A. `time_forward` (default) — recommended for realistic forecasting / sim-to-future evaluation:

   * Sort by ts. Reserve the **last** `test_span_days` as test, the immediately preceding `val_span_days` as validation, and the earlier portion as training. If dataset spans multiple households, apply same calendar cut to all households so test simulates future for all nodes. Ensure `test` contains at least `--min_event_count` of each rare event; if not, shift windows or greedily move event occurrences from train to test with minimal temporal disturbance (log changes).

   B. `block_hierarchy` — preserve contiguous episodes (useful if there are non-overlapping episodes):

   * Identify episode boundaries (e.g., long gaps > 24h per household) and split episodes into train/val/test blocks with configurable ratios, ensuring whole episodes go to one split.

   C. `episode_random` — random split by episodes (when episodes are independent):

   * Randomly assign episodes to splits stratified by event_flag distributions and household archetype.

   D. `kfold_ts` — time-aware K-fold cross-validation:

   * Create K folds of contiguous time windows (expanding-window or rolling-window designs). Provide utilities to generate train/val/test for each fold.

   Implementation details for splitting:

   * Always preserve full rows for a given `(household_id, ts)` — do not split within the same timestep.
   * When moving a small number of rows to meet `min_event_count` constraint, choose the least disruptive rows (e.g., move last occurrence in train to test). Log every manual relocation into `split_adjustments.log`.
   * Save split assignments as `split_index.parquet` with columns: `ts,household_id,split_label` and a checksum.

9. **Stratification & balancing**

   * Provide optional stratification keys: `event_flag`, `household_archetype` (derived from config), and `time_of_day_bucket`. When `--strategy` is `episode_random` or `kfold_ts`, ensure stratified assignment so test set reflects rare events and archetype diversity.
