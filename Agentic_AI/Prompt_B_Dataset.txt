Core functional requirements (detailed per-timestep loop — required):

For each household and each timestamp (`ts`) in the simulation window do:

1. **Input selection / measured-mode check**

   * If collected includes `household_id` matched to this household and measured `pv_gen_kw`/`load_kw` available for `(household_id, ts)`, **use measured values**.
   * Otherwise compute `pv_gen_kw` using `pvlib.get_total_irradiance` with panel tilt/azimuth + `pvlib.pvsystem.pvwatts_dc` (or similar) and `system_efficiency`. Use `tz` from collected metadata for solar position calculations. `pv_gen_kw` = (POA_irradiance / 1000) * panel_kwp * efficiency.

2. **Load synthesis**

   * If `load_kw` measured, use it. Else synthesize using `load_template` scaled by household archetype and time-of-day (weekday/weekend). Templates should be included (3 defaults) and scaled to Indian magnitudes.

3. **Compute `net_kw`**

   * `net_kw = pv_gen_kw - load_kw`. Store numeric with float precision.

4. **Battery control & SoC update (default conservative policy)**

   * Decide charge/discharge action using a configurable rule (sell if surplus > threshold and soc above margin; charge when cheap/solar; discharge during high price or deficit).
   * Compute `battery_power_kw` (positive charge, negative discharge), limited by `battery_nominal_power_kw`.
   * Update SoC:

     ```
     if charging: soc += (battery_power_kw * timestep_hours) * efficiency_charge
     if discharging: soc -= (abs(battery_power_kw) * timestep_hours) / efficiency_discharge
     ```
   * Clip SoC to [soc_min, soc_max]. If clipping occurred due to attempted action, set `safety_violation=True` for that row and log the event.

5. **Offers creation**

   * If policy decides to sell surplus and `net_kw > min_sell_threshold`, create an `offer`:

     * `offer_id` unique (e.g., `offer_<household>_<ts_unix>`).
     * `offered_kwh` = min( net_kw * timestep_hours, max_offer_kwh, soc_kwh - soc_min ).
     * `offer_price` strategy: configurable (e.g., utility_price × markup or dynamic).
   * For the offer row, compute `commit_hash`:

     * Exact concatenation and hashing method:
       `commit_hash = sha256(f"{ts}|{household_id}|{offered_kwh:.6f}|{salt_hex}")`
       where `salt_hex` is hex-encoded random salt of length from config.
     * Save salt to `secrets/household_id/<offer_id>.salt` (file mode `600`) or to an encrypted store if `--encrypt-salts` used.

6. **Market engine & matching**

   * Maintain open offers in neighborhood (configurable neighbor list/radius).
   * Matching rules (configurable):

     * `price_below_utility`: buyers accept if `offer_price <= utility_price_at_ts`.
     * `nearest_neighbor`: match any acceptor in radius with capacity.
     * `probabilistic`: buyer accepts with probability depending on price gap.
   * When matched: generate `trade_id`, link `offer_id`, set `matched_trade_id` in offers, and mark `trade_id` in seller & buyer rows.

7. **Settlement simulation**

   * After delivery window, simulate settlement:

     * Verify delivered energy (for simulation use delivered = offered_kwh ± small noise) and check `abs(delivered - offered_kwh) <= tolerance`.
     * If verified, mark `settled_bool=True`, `settle_ts` = current timestamp, compute `amount_paid` = `delivered * offer_price`, and generate a `settlement_tx_stub` (random hex string).
     * If verification fails, `settled_bool=False` and log dispute reason.

8. **Reward computation (if `--compute-reward`)**

   * Compute per-timestep reward using config coefficients:

     ```
     reward = cost_savings + grid_stability_bonus - battery_degradation_cost
     ```

     * `cost_savings`: energy sold * (offer_price - utility_price) + energy avoided from grid * utility_price.
     * `grid_stability_bonus`: negative of neighborhood imbalance (abs) scaled by coefficient.
     * `battery_degradation_cost`: proportional to energy cycled (kWh) times `battery_deg_coeff`.
   * Store `reward` (float) in generated dataset row.

9. **Logging & event flags**

   * Set `event_flag` values for injected scenarios (`cloud_ramp`, `ev_arrival`, `grid_outage`, `normal`).
   * Record any `safety_violation` boolean.

10. **Row writing**

    * Populate `generated_dataset` row with all canonical fields and write/stream to output.

Noise, scenarios & robustness (required):

* Add Gaussian noise to synthesized `pv_gen_kw` and `load_kw` as per `noise` config.
* Inject missing readings at rate `missing_rate` (config).
* Cloud ramps: randomly sampled by `cloud_ramp_prob_per_day`; when triggered, scale PV down by severity (configurable) for a random duration. Mark `event_flag='cloud_ramp'`.
* EV arrival: sudden load spike event with configured probability and magnitude. Mark `event_flag='ev_arrival'`.
* Outages: simulate grid isolation when `outage` event triggers; in outage, buyer-of-last-resort behavior changes accordingly.

Quality checks & validations (must run; fail non-zero on critical issues):

* **Header check:** generated CSV headers must exactly match canonical header above (case-sensitive).
* **Mass balance:** `abs(net_kw - (pv_gen_kw - load_kw)) < 1e-6` for every row. If violated, auto-correct `net_kw` to `pv_gen_kw - load_kw` and log correction.
* **SoC bounds:** `soc_kwh` always within [soc_min, soc_max]; if clipped set `safety_violation=True` and count.
* **Offer/trade integrity:** each `trade` must reference an existing `offer_id`; every `offer` created must have a `commit_hash`.
* **Salt leakage check:** scan CSVs and logs before final write to ensure no salt hex strings are present.
* **Determinism:** same `--seed` + config must produce same outputs; include a reproducibility test in unit tests.
