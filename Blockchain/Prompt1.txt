## Goals (high level)

1. Pi generates secp256k1 keypair and stores private key in secure element (ATECC608A/TPM) or encrypted keystore fallback.
2. `node_id = keccak256(pubkey | salt)` is derived locally. Pubkey or pubkey_hash and `metaURI` (off-chain profile) are registered to `IdentitySC` contract on an L2 dev network (Hardhat/Anvil).
3. Off-chain PII/KYC stored in a permissioned Postgres registry; only `metaURI` (IPFS CID or HTTPS) or hash is stored on-chain.
4. Device attestation step (simulate ATECC cert check; if unavailable, simulate TPM attestation token).
5. Provide relayer (meta-tx) so Pi can register without native gas tokens.
6. Provide tests: unit tests for contract, integration test for Pi â†’ relayer â†’ IdentitySC, and an HIL test simulation for a Pi (or Docker container).

### 2) Smart contract: IdentitySC (Solidity skeleton)

Create `contracts/IdentitySC.sol` with these minimal APIs:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.18;

import "@openzeppelin/contracts/access/AccessControl.sol";

contract IdentitySC is AccessControl {
    bytes32 public constant ADMIN_ROLE = keccak256("ADMIN_ROLE");

    struct Node {
        address owner;      // pubkey/eth address
        bytes32 pubkeyHash; // keccak256(pubkey | salt) or keccak(pubkey)
        string metaURI;     // IPFS CID or HTTPS to off-chain profile (PII stored off-chain)
        uint256 stake;
        uint256 registeredAt;
        bool active;
    }

    mapping(bytes32 => Node) public nodes; // keyed by node_id (bytes32)

    event NodeRegistered(bytes32 indexed nodeId, address indexed owner, string metaURI);
    event NodeRevoked(bytes32 indexed nodeId);

    constructor(address admin) {
        _setupRole(DEFAULT_ADMIN_ROLE, admin);
        _setupRole(ADMIN_ROLE, admin);
    }

    function registerNode(bytes32 nodeId, bytes32 pubkeyHash, string calldata metaURI) external {
        require(nodes[nodeId].registeredAt == 0, "node exists");
        nodes[nodeId] = Node(msg.sender, pubkeyHash, metaURI, 0, block.timestamp, true);
        emit NodeRegistered(nodeId, msg.sender, metaURI);
    }

    function revokeNode(bytes32 nodeId) external onlyRole(ADMIN_ROLE) {
        nodes[nodeId].active = false;
        emit NodeRevoked(nodeId);
    }
}
```

### 6) Pi key generation & secure storage

Provide two flows:

**A. Hardware secure element (preferred)**

* Use ATECC608A tooling: provisioning script to create key in the device. Store private key only in secure element.
* The Pi script gets the public key from the secure element and computes `node_id = keccak256(pubkey || salt)` where `salt` is random device salt stored in secure element or encrypted file accessible only to device.

**B. Software fallback (encrypted keystore)**

* Use `ethers.Wallet.createRandom()` to generate keys.
* Encrypt with a strong passphrase and store in `/etc/gridguardian/keystore.json` with POSIX 600.
* Example generate script (pi-client/generate_keys.js):

```js
const { ethers } = require("ethers");
const fs = require('fs');
async function main() {
  const wallet = ethers.Wallet.createRandom();
  const password = process.env.KEYSTORE_PW || "change_this";
  const encrypted = await wallet.encrypt(password);
  fs.writeFileSync('/home/pi/.gridguardian/keystore.json', encrypted, {mode: 0o600});
  console.log("pubkey:", wallet.address);
}
main();
```

**Salt & node_id**

* Generate 32-byte random salt on device and store locally (POSIX 600).
* Node id: `node_id_bytes = keccak256(abi.encodePacked(pubkey_bytes, salt_bytes))` â†’ hex string (32 bytes). Keep salt private.

### 8) Relayer: meta-tx registration (so Pi neednâ€™t hold native chain token)

* Relayer runs a simple service (`relayer/server.js`) that accepts signed EIP-712 messages authorizing `registerNode` with parameters. Relayer sends the actual on-chain transaction paying gas, and optionally charges stablecoin later.
* Relayer verifies signature and nonce, constructs the tx and calls `IdentitySC.registerNode(...)` from relayerâ€™s account.

Sample flow for Pi:

1. Pi signs typed data (EIP-712) that contains: `nodeId`, `pubkeyHash`, `metaURI`, `nonce`, `expiry`.
2. Pi sends signature + payload to `relayer/api/register`.
3. Relayer verifies signature, then sends a tx to the chain: `IdentitySC.registerNode(nodeId, pubkeyHash, metaURI)` and attaches `msg.sender = relayer` (contract uses `msg.sender` as owner address â€” to keep owner as device, you can use meta-tx pattern to set owner param to device address).

Implement EIP-712 typed data example in `pi-client/register_via_relayer.js`.

## Security & best practices (must follow)

* **Never** store salts or private keys in plain text in the repo. Use OS file permissions (600) and/or secure module.
* Use EIP-712 for typed signing and replay protection (nonces + expiry).
* On the chain, store only `node_id` and `metaURI` (or `metaHash`). Keep PII off-chain.
* Device attestation is required for production; in dev simulate attestation but design interfaces identical to production.
* Use HTTPS + mTLS for relayer and registry endpoints.
* Use timelocks & admin role for revocations.
